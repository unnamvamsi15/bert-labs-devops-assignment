1. Architecture Overview & Best Practices

This document outlines the deployment strategy for a cloud-native web application consisting of a React Frontend and a Node.js (Express) Backend. As a Junior DevOps Engineer, your goal is to follow these steps precisely to achieve a production-ready environment.

Core Principles Applied:

  • Security: Principle of least privilege (non-root containers, IAM roles), encrypted state files, and secrets management.
  • Scalability: Horizontal Pod Autoscaling (HPA), Cluster Autoscaler, and Stateless application design.
  • Reliability: Multi-AZ (Availability Zone) deployments, readiness/liveness probes, and rolling updates with zero downtime.
  • Automation: Everything as Code (IaC, CI/CD). No manual changes in production.

2. Containerization (Docker)

We use Docker to package the application. Both Dockerfiles implement multi-stage builds to minimize the final image size and reduce the attack surface.

2.1 Backend Dockerfile

Create a file named Dockerfile inside the backend/ directory.

# Use specific LTS Alpine image for security and size
FROM node:18-alpine

# Set Node environment to production
ENV NODE_ENV=production

# Create app directory and set ownership to node user
WORKDIR /usr/src/app
RUN chown -R node:node /usr/src/app

# Switch to non-root user (Security Best Practice)
USER node

# Install dependencies first (leverage Docker cache)
COPY --chown=node:node package*.json ./
RUN npm ci --only=production

# Copy application source code
COPY --chown=node:node . .

# Expose the port the app binds to
EXPOSE 3001

# Start the application
CMD [ "node", "server.js" ]
Why this is production-ready: We use npm ci for deterministic builds. Running as the node user instead of root prevents privilege escalation attacks if the container is compromised.

2.2 Frontend Dockerfile

Create a file named Dockerfile inside the frontend/ directory.

# Stage 1: Build the React application
FROM node:18-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Serve the app with Nginx
FROM nginx:1.25-alpine
# Copy built assets from the build stage
COPY --from=build /app/build /usr/share/nginx/html
# Expose port 80
EXPOSE 80
# Run Nginx in foreground
CMD ["nginx", "-g", "daemon off;"]

3. Infrastructure as Code (Terraform & AWS EKS)

We provision our infrastructure using Terraform. For this playbook, we target AWS and Elastic Kubernetes Service (EKS).

3.1 Remote State Configuration (Security & Collaboration)

Never store Terraform state locally. Create a provider.tf file:

terraform {
  backend "s3" {
    bucket         = "company-production-terraform-state"
    key            = "app/production/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks" # Prevents concurrent modifications
    encrypt        = true
  }
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

3.2 Provisioning the EKS Cluster

Create a main.tf file. We utilize the official AWS EKS module for best practices.

module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"

  name = "production-vpc"
  cidr = "10.0.0.0/16"
  azs  = ["us-east-1a", "us-east-1b", "us-east-1c"]
  
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = false # High Availability
}

module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 19.0"

  cluster_name    = "production-cluster"
  cluster_version = "1.28"

  vpc_id                   = module.vpc.vpc_id
  subnet_ids               = module.vpc.private_subnets
  control_plane_subnet_ids = module.vpc.public_subnets

  eks_managed_node_groups = {
    app_nodes = {
      min_size     = 2
      max_size     = 10
      desired_size = 3
      instance_types = ["t3.medium"]
    }
  }
}

Execution Steps: Run terraform init, followed by terraform plan, and finally terraform apply.

4. Container Orchestration (Kubernetes)

Once EKS is up, we deploy our application using Kubernetes manifests. Create a directory named k8s/.

4.1 Backend Deployment & Service (k8s/backend.yaml)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
  namespace: production
spec:
  replicas: 3 # High Availability
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
      - name: backend
        image: your-registry.com/backend:latest
        ports:
        - containerPort: 3001
        # Resource Limits (Crucial for stability)
        resources:
          requests:
            cpu: "100m"
            memory: "256Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
        # Security Context
        securityContext:
          runAsNonRoot: true
          allowPrivilegeEscalation: false
        # Probes ensure traffic only routes to healthy pods
        readinessProbe:
          httpGet:
            path: /
            port: 3001
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /
            port: 3001
          initialDelaySeconds: 15
          periodSeconds: 20
---
apiVersion: v1
kind: Service
metadata:
  name: backend-service
  namespace: production
spec:
  selector:
    app: backend
  ports:
    - protocol: TCP
      port: 80
      targetPort: 3001

4.2 Horizontal Pod Autoscaler (HPA)

To ensure scalability during traffic spikes, apply an HPA for the backend.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: backend-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
Action: Apply manifests using kubectl apply -f k8s/. Ensure you have created the namespace first: kubectl create namespace production.

5. Continuous Integration & Deployment (CI/CD)

We utilize GitHub Actions to automate testing, building, and deployment.

5.1 GitHub Actions Workflow (.github/workflows/deploy.yml)

name: Production Deployment

on:
  push:
    branches:
      - main

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v3

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v1

      - name: Build, tag, and push Backend Image
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          ECR_REPOSITORY: backend
          IMAGE_TAG: ${{ github.sha }}
        run: |
          cd backend
          docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG

      - name: Update KubeConfig
        run: aws eks update-kubeconfig --name production-cluster --region us-east-1

      - name: Deploy to Kubernetes
        run: |
          # Replace image tag dynamically in deployment
          sed -i "s/latest/${{ github.sha }}/g" k8s/backend.yaml
          kubectl apply -f k8s/backend.yaml -n production

6. Monitoring & Observability

A production system is blind without monitoring. Implement the following stack:

  • Metrics: Install Prometheus and Grafana using the official Helm charts.
  • Command: helm install kube-prometheus-stack prometheus-community/kube-prometheus-stack -n monitoring --create-namespace
  • Logs: Deploy Fluent Bit as a DaemonSet to forward container logs to Elasticsearch/OpenSearch.
  • Alerting: Configure Alertmanager (bundled with kube-prometheus-stack) to send Slack or PagerDuty alerts if CPU > 85% or Pods are crashing.

7. Operational Troubleshooting Guide

As a DevOps Engineer, you will face incidents. Follow these precise steps to diagnose and resolve common issues.

Symptom / Error Root Cause Analysis Steps Resolution Approach
CrashLoopBackOff
(Pod keeps restarting)		 1. kubectl logs <pod-name> -p (Check previous container logs).
2. Look for application errors (e.g., missing ENV vars, database connection refused). 		Fix the application code, update the ConfigMap/Secret, and trigger a rolling restart: kubectl rollout restart deployment backend.
ImagePullBackOff / ErrImagePull 1. kubectl describe pod <pod-name>
2. Check the Events section at the bottom. 		Verify the image tag exists in ECR. Ensure the Kubernetes nodes have the correct IAM role permissions to pull from the registry.
Pending Pods 1. kubectl get events --sort-by='.metadata.creationTimestamp'
2. Usually caused by insufficient CPU/Memory in the cluster. 		Check Cluster Autoscaler logs. Scale up the EKS Node Group manually if autoscaling is failing, or adjust Pod Resource Requests.
502 Bad Gateway / 503 Service Unavailable 1. Check if Endpoints exist: kubectl get endpoints backend-service.
2. If none, check if Pods are passing their Readiness Probes. 		If pods are running but not ready, the app might be deadlocked. Check app logs. If endpoints exist, check Ingress Controller logs.
Final Rule for Junior Engineers: When in doubt, read the logs. Do not execute destructive commands (like deleting PVCs or state files) without consulting a Senior Engineer. Automation is your friend; manual fixes in production are your enemy.
